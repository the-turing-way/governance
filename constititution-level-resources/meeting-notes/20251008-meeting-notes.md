## Notes
* Reviewing decision making process
  * Where do community members open SC items? Discussion or issues
  * Governance repo issues is harder to find than discussions
  * Governance repo may seem exclusive or out of bound for community members
  * I like discussions - it's more explicitly open for the community AND it's in line with what we're nudging the community towards +3
  * Community is moving towards using discussions for anything but issue-like-tasks
* Issues for votes,
  * Works conceptually as there is a task here, SC votes, it is done when there is a decission
  * Could be organised in a project board for convenience
  * Votes in discussions could become difficult to find, amongst other discussions, somewhere in a thread
* Issue template could help make this easier to set up
* **Consensus is:** Use discussions for proposals, discussion, gauging community feelings. When ready for a vote, open a voting issue (in governance repo) 
* Pushing back to maintenance level
  * Is this a vote, or an informal process?
  * Requiring SC to vote every time there is a question whether they need to be involved could generate a lot of work, delays
  * Potentially problems giving SC members the ability to unilaterally push things back to maintenance level. What if there is disagreement?
    * Maybe not such a problem. These decisions are not permanent.
  * This is a bit like issue triage, we trust individuals to do this as representatives of the community
* Are we all able to categorise SC material and maintenance material?
  * Keep focus on empowering community. "We don't ask to be informed so we can manage what you do, but so we can help enable you"
  * Make sure guidance, documentation is clear that we want to empower people. They should feel able to do things and not feel that they must get approval
  * Push people to ask **the community**, not the steering committee
* **Proposal:** Leave the community to decide. If the community cannot agree, this becomes a vote for the SC
